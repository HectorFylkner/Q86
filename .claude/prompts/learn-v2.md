# Q86 Learn v2 — make the learning segment the best in the world for one student

## Mission

Q86 is a local-first, single-user training platform for GMAT Focus Quantitative Reasoning; the name is the target score. Your job is to develop its learning segment — the lessons, chapter tests, mastery model, and every loop connecting them to practice — until it is a demonstrably stronger learning system than Target Test Prep or Manhattan Prep for its one user: a motivated adult optimizing score gained per hour spent.

The competitors define the opening. TTP is the market's best-reviewed quant course, and its own structure is the weakness: ~180 prescribed hours for quant alone, personalization at calendar level only (every student marches the same lesson sequence, no test-out), retention left entirely to user discipline (students report forgetting chapters 1–5 by chapter 20), and pedagogy that is overwhelmingly passive reading. Manhattan's strength — strategy woven into content so students learn *when* to use each method, not just the math — comes with too little drill volume and explanations too shallow for a Q86 aspirant. Q86's course already reads in ~3 hours and out-instruments both products on diagnosis. What it lacks is closure: signals are collected and dropped. You are mostly connecting a product that already exists — then deepening content exactly where evidence says depth wins.

Four laws govern every design decision. When a choice the workstreams below don't settle comes up, decide by these:

1. **No signal dead-ends.** Every diagnostic the system produces — a miss, an error tag, a coach attribution, a stale mastery cell — terminates in a scheduled, one-tap action, never in a wall of analytics.
2. **Nothing is invisible to the scheduler.** All learning state lives where the daily plan can read it. A feature that stores progress the plan can't see is mis-built.
3. **Retrieval beats rereading.** Wherever the user could passively re-encounter content, make them produce something instead: an answer, a strategy commitment, a recalled cue. This is the single best-evidenced principle in the learning-science literature and the deepest flaw in every competitor.
4. **Respect the user's hours.** Evidence of existing mastery skips content; nothing is hard-locked; density is a feature. Add mechanics and chapters, never words.

Treat the acceptance criteria below as floors, not ceilings — where you see a higher-leverage move consistent with these laws, take it.

## The system you are extending

Read these before designing; the details below are orientation, not a substitute.

**The learning segment today.** 24 markdown chapters in `content/lessons/<subtopic>.md`, one per taxonomy subtopic, ~1,400 words each, all conforming to a strict seven-section dialect (exact `##` headings: Why this matters / The core ideas / Worked examples / Trigger cues / Trap gallery / Speed moves / Before you drill) parsed by `lib/lesson-parse.ts` — numbered ideas, exactly 3 worked examples (`**Example N**`, italic-wrapped question, numbered steps, `**Answer: …**`), arrow-split cues, titled trap/speed bullets, a 7-item first-person checklist. On any deviation the parser returns null and `app/learn/[subtopic]/page.tsx` falls back to the generic `<Md>` renderer (`components/math.tsx` — a deliberately constrained dialect: inline/display KaTeX, bold/italic/code, lists, headings; no HTML, no images, no links). Chapter tests (`lib/chapter-tests.ts`, config in `lib/chapter-test-config.ts`): 8 questions, 75% bar, difficulty blend 2/3/2/1 across D2–D5, run through the ordinary drill runner; pass state is derived by scanning completed sessions and is sticky. Mastery ladders (`lib/mastery.ts`): per-subtopic rungs D2–D5, mastered at ≥85% over the most recent 10 focused attempts (min 6), advisory only. Lesson read/checklist state lives solely in localStorage (`q86-learn:<subtopic>`).

**The central defect.** Lessons, chapter tests, and mastery ladders are three progress models that never talk — and none of them feeds `lib/plan.ts`, whose foundations-phase copy literally tells the user to "climb the mastery ladders" while `computeDailyPlan` cannot see them. Meanwhile the diagnosis pipeline dead-ends three ways: the post-mortem coach's drill prescription renders as inert text, `content_gap` misses never link to the chapter that fixes them, and `attempts.error_subtag` (concept-level miss attribution) is written but read by nothing.

**Adjacent systems and their hook points.** The redo queue (`lib/redo.ts`) auto-enrolls misses at +2d/+7d/+21d with a 150-second cold-solve gate — question-scoped only. The flashcard deck (`lib/deck.ts`, pure scheduler in `lib/srs.ts`) cuts cards only from missed questions' solutions, keyed by questionId, 12-card daily cap. The deterministic daily plan splits pure logic (`lib/plan.ts`) from data gathering (`lib/plan-server.ts` → `gatherPlanInputs`). Question selection (`lib/engine.ts` → `selectQuestions`) down-weights twice-correct questions to 0.15. The error taxonomy (`lib/taxonomy.ts`) is the single spine: subtopic key = lesson filename = route param = question column = coach attribution = analytics dimension = official score-report category. Attempts carry confidence (guess/lean/lock), time, error type, and session focus (casual attempts are excluded from all performance statistics). The bank is ~360 verified questions with thin cells — `interest_profit_discount` has 8 total; D5 across the whole bank is ~72.

## Workstreams

In build order, which is also roughly score-impact-per-effort order. Each is a coherent, independently shippable increment.

**0 — Authoring guardrails (first; everything content-related depends on it).**
A `scripts/validate-lessons` script in the style of `scripts/verify-bank.ts`: runs `parseLesson` over every file in `content/lessons/`, asserts non-null plus the section minimums, prints per-file diagnostics, exits non-zero on failure. Fix the lesson header's hardcoded "3 worked examples" to count `parsed.examples.length`.
*Done when:* the validator passes on all 24 current files, fails loudly on a deliberately broken copy, and is documented alongside the question harness in the README's authoring section.

**1 — One progress spine: the curriculum joins the daily plan.**
Persist lesson progress server-side (a small table or settings row: readAt, checklist state per subtopic — migrate or dual-read the existing localStorage keys). Feed lesson progress, `chapterTestStates()`, and `computeLadders()` into `gatherPlanInputs`, and give `DailyPlan` a curriculum slot: next chapter = weakest unpassed subtopic by drill evidence, baseline report, and ladder gaps. On the learn index, order chapters weakest-first within skill groups and mark chapters the evidence says are already known with a test-out CTA straight to the chapter test (`/drill?test=X` already works).
*Done when:* a fresh database yields a plan that leads with study; strong drill history in a subtopic surfaces test-out on its chapter; the plan's curriculum card deep-links to `/learn/<subtopic>`; TTP's signature failure — a strong student grinding low-yield basics in a fixed sequence — is structurally impossible here.

**2 — Close the remediation loop.**
The post-mortem's drill prescription becomes a one-tap "Start prescription" (via the existing `startDrillWithQuestions`/`selectQuestions`). `content_gap` misses and coach concept attributions render a "Reread: <chapter> → <section>" deep link with a section anchor. Analytics heatmap cells and the subtopic×difficulty matrix link to `/learn/<subtopic>` and `/drill?sub=&d=`. `error_subtag` gets consumed: a cross-attribution view ("misses filed under X that were really Y") that also nudges the plan's curriculum slot.
*Done when:* no surface in the app names a subtopic without linking to its lesson and its drill, and a `content_gap` tag recorded today changes what the plan schedules tomorrow.

**3 — Retention: concept-level spaced retrieval and interleaved review.**
Two mechanics, both scheduled — never left to discipline. (a) On chapter-test pass, enroll that chapter's trigger cues and trap-gallery entries as retrieval-first cards ("you see: <phrase> — what do you do?" → reveal) in a lesson-scoped review table shaped like the existing deck reviews, scheduled by the pure `lib/srs.ts`, merged into the daily deck under its cap with question-derived cards keeping priority. (b) Mixed, interleaved question sets become the plan's default drill block once a subtopic exits first acquisition (keep blocked drilling for the first sets after a lesson): a selector weighting staleness × weakness across all studied subtopics, difficulty centered on each ladder's current rung, with a small hand-authored confusable-pairs table (~8 pairs: permutation/combination, rate/ratio, and kin) co-scheduling both members. Target ~70–85% success in mixed review.
*Done when:* a chapter passed in week 1 resurfaces in week 4 as cue cards and mixed-set questions with zero user action, and the plan's post-acquisition drill block spans subtopics instead of blocking on one.

**4 — Honest mastery: decay, pace, and test integrity.**
Mastery gains two missing dimensions. Staleness: a mastered cell whose newest attempt is >21 days old shows stale and re-enters review weighting until a short confirmation set re-proves it. Pace: mastered additionally requires median time over the window under a per-difficulty bar (roughly 2:00 at D3 scaling to 2:45 at D5 — tune against the exam's 45 min/21 questions); accurate-but-slow renders as its own state with a timed-drill CTA. Chapter-test integrity: `chapterTestStates()` recomputes correct/total from attempts rows joined on sessionId instead of trusting the client summary; retakes exclude questions attempted in the last 14 days; a subtopic that cannot field 8 eligible questions says "bank too thin — N/8" instead of running a shrunken test; raw scores ("6/8") render wherever a pass badge appears.
*Done when:* at the Q86 level, where the binding constraint is pacing on D4/D5, mastery states answer both "still true?" and "fast enough?" — and a pass badge can no longer mean 3/4.

**5 — Active lessons: the honor system becomes evidence.**
Worked-example reveal currently costs one tap and captures nothing. Before reveal, require a strategy commitment (algebra / backsolve / smart numbers / test cases) and an answer commitment (A–E tap, or a one-line numeric when there are no choices); log both server-side with timing. Reveal then shows the user's pick against the answer. The learn index's "prepared" badge becomes evidence-based (examples attempted, checklist ticked) rather than checkbox-based, and lesson-example attempts join analytics so read→drill lift is measurable for the first time. Keep the try-before-reveal interaction pattern — this upgrades it, it does not replace it.
*Done when:* all 72 worked examples are retrieval events with logged commitments, and the strategy taps produce per-method accuracy/time data.

**6 — Content deepening: strategy chapters, bank depth, visual methods.**
(a) Author two new chapters in the exact existing dialect, placed on the learn index: **Data Sufficiency decision discipline** — the AD/BCE elimination tree, value vs yes/no questions, statement-isolation hygiene, with the trap gallery built from sufficiency-specific wrong turns (DS is a first-class format in the taxonomy yet no chapter teaches its logic) — and **Choosing the fastest path** — the same problem solved two or three ways in the worked examples with the timing comparison as the payoff, turning Manhattan's strategy-selection strength into a trainable, measured habit via workstream 5's strategy taps. (b) Run a targeted authoring campaign through the existing gate (`scripts/author/harness.mjs` — every item's `check()` recomputes the key by brute force; then `verify-bank`, then `pnpm seed`): raise every subtopic to a floor of ~25 verified questions with ≥8 at D4–D5, thinnest and weakest subtopics first. Surgical depth, not TTP's 3,000-question grind. (c) Add two or three named inline components to the Md dialect as constrained text directives — a 2×2 set matrix, a number line with labeled zones, a rate/work timeline — and deploy them in the handful of chapters where the picture is the method (overlapping sets, absolute value/number line, inequalities, rates, mixtures). Named components only; no general image or HTML support.
*Done when:* both chapters parse under the validator and read at the density of the best existing ones; no subtopic can exhaust its D4/D5 pool inside one review cycle; the matrix and number-line methods are shown, not described.

## Invariants

- **The taxonomy spine is sacred.** Every new feature joins on the subtopic key. Never introduce a parallel category namespace.
- **The question gate is absolute.** No question reaches the database without a programmatic `check()` through the authoring harness plus `verify-bank`. LLM agreement alone is never sufficient. This is the platform's signature guarantee.
- **The lesson dialect is a contract.** Parser, validator, and content move in lockstep; the generic-Md fallback keeps rendering anything that doesn't parse; dialect extensions degrade gracefully.
- **Derived, not stored.** Progress is recomputed from attempts and sessions wherever possible. New state is limited to what genuinely cannot be derived: read timestamps, card scheduling, example commitments.
- **Advisory, never locked.** Sequencing is recommendation-strength everywhere. Hard gates are TTP's most-resented feature; the ladders' "tells you where to work, does not forbid working elsewhere" stance extends to everything you build.
- **Do not regress what is already superior:** per-attempt confidence tagging and the calibration dashboard; the scratch-work post-mortem loop with twin generation; the redo queue's cold-solve gate and auto-enrollment; the review-and-edit ledger and the decide (solve/guess/bail) trainer; lesson density (~7-minute chapters); the try-before-reveal cards; casual-session exclusion from statistics. No commercial product has these. They are load-bearing inputs to your work, not legacy to simplify away.
- **Prohibited (over-build for n=1):** IRT/BKT or knowledge-component graphs (population-scale tools; windowed accuracy + decay + pace captures the value); video content; accounts, sync, or cloud state; anti-cheat beyond server-side recomputation; A/B-testing infrastructure (pick literature-informed constants and expose them as named config); a predicted-score widget (official mocks are the calibration layer); LLM grading on every attempt (keep it in the post-mortem coach; use tap-based commits elsewhere); a general diagram/image pipeline.

## Engineering conventions

Match the repo exactly; it is unusually consistent. Pages are async server components exporting `dynamic = "force-dynamic"` and `runtime = "nodejs"`, querying Drizzle directly, delegating interactivity to a single `components/<feature>/<feature>-client.tsx`; `searchParams` is a Promise. Mutations are server actions in `lib/actions.ts` returning `{ error: string | null, … }` — never thrown to the client; client flow state is a discriminated-union `Stage`. Imports: `lib/` and `scripts/` use relative paths with explicit `.ts` extensions (they run under Node type stripping); `app/` and `components/` use the `@/` alias without extensions. Enums live in `lib/taxonomy.ts` as `const … as const` arrays with derived types; DB columns brand to them. Styling uses token utilities only (bg-paper/surface, text-ink/graphite/ballpoint/redpen, border-grid, rounded-card, shadow-ambient, font-display/font-mono) — no raw hex; style both dark-mode selector forms; numbers in font-mono. Rich text renders through `<Md>`; escape currency as `\$`. Schema changes are a triple: edit `lib/db/schema.ts`, add the migration under `drizzle/`, and mirror it as guarded idempotent DDL in `lib/db/bootstrap.ts` `evolveSchema()`. Rows are never deleted — retire with flags so attempt history survives. Substantial why-comments and JSDoc on exports; complete-sentence user copy; no emojis in UI.

## Verification and delivery

There is no test suite; verification here is mechanical and behavioral. After every workstream: `pnpm lint`, `pnpm build`, the lesson validator, and — after any bank change — `verify-bank` then `pnpm seed` (which must stay idempotent). Then exercise the loop end-to-end against a dev server with a seeded database: read a chapter, commit on its examples, pass its test, confirm the cards enroll, the plan re-sequences, the analytics links resolve, and a tagged `content_gap` miss round-trips into tomorrow's plan. Check new UI in light and dark, and with both PS and DS questions.

Work autonomously through the workstreams in order; each lands as one or more lint-clean commits with clear messages, so any later session can resume from `git log` alone. When a workstream's acceptance criteria are met, verify against them explicitly before moving on — report what you can point to evidence for, and label anything unverified as unverified. Stay inside the learning segment and its integration seams; adjacent systems are hook points, not refactor targets. If a genuine fork arises that the four laws and invariants cannot settle — a schema choice that constrains future work, or a conflict between an acceptance criterion and something you discovered in the code — surface it with a recommendation rather than guessing silently. Otherwise: decide by the laws, build, verify, commit, continue.
